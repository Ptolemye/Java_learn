# 2. 内存结构

## 2.1 程序计数器

- 采用寄存器结构，可以记住下一条JVM指令==(字节码)==执行地址的行号
- 每个线程都有自己==私有==的PC，各管各的
- PC不存在内存溢出问题



## 2.2 栈

**STACK**

==**线程**==运行所需要的内存空间，JVM会为每一个==**线程**==开辟一个虚拟机栈

**栈帧**：每个**方法**运行时所需要的内存，其中就包括了参数，局部变量，返回地址

**活动栈帧**：当前线程正在执行的那个方法所对应的栈帧。

**Q1 垃圾回收会管理栈内存**

每一Q个方法调用结束后，栈帧自动出栈，不需要垃圾回收机制来回收

**Q2 方法内的局部变量是否线程安全**

- 安全，每个线程都有自己独立的栈空间，局部变量存储在栈空间的栈帧中，线程与线程之间互不影响。
- 如果是局部变量引用了对象（局部变量只是引用，实际的对象存储在堆中），并逃离了方法的访问，那就要考虑线程安全问题。(==逃逸==)。（1）变量必须是定义在方法内部的局部变量（2）变量如果是引用，没有逃离方法的作用范围

**Q3 栈内存分配是不是越大越好**

不是，物理内存是有限的，单个栈分配的内存越多，可以同时产生的线程就越少



**栈内存溢出**

（1）栈帧太大（定义了太多局部变量，很罕见），太深（==递归调用==，栈帧数量太多），都有可能导致栈内存溢出

（2）代码逻辑可能没有涉及无限递归，但是类/方法之间可能互相调用导致栈内存溢出



**线程运行诊断**

（1）CPU占用过高

- 用 top 定位哪个进程对 cpu 的占用过高
- `ps H -eo pid,tid,%cpu | grep 进程id`
   用 ps 命令进一步定位是哪个线程引起的 cpu 占用过高
- `jstack 进程id`
  - 可以根据线程 id 找到有问题的线程，进一步定位到问题代码的源码行号

（2）死锁问题



**本地方法栈**

`native`关键字修饰的方法底层都是用C语言实现的

Java 虚拟机栈：给 Java 方法用
本地方法栈：给 native 方法用

**本地方法栈就是 JVM 为 `native`方法准备的线程私有栈空间，用来支持 Java 调用操作系统或 C/C++ 等本地代码。**
